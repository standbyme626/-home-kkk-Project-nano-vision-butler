# Test Necessity Analysis (RK3566 Edge)

## 结论
- 必须测试，而且必须分层测试。
- 只测“模型 FPS”无法证明系统满足项目目标。

## 为什么“必须测”

### 1. 模型可跑 != 业务可用
- 模型层只回答“推理快不快、准不准”。
- 你的项目目标还包含事件上报、状态查询、命令闭环、审计追踪，这些都依赖系统链路协同。

### 2. 边缘端是受限环境
- RK3566 属于资源受限设备，2G 内存下对模型体量、缓存、进程稳定性都敏感。
- 不做稳定性测试，容易出现“短测能跑、长跑崩溃”的假通过。

### 3. 不同模型族后处理并不通用
- 主检测候选跨 YOLOv5/v6/v8，输出形态和后处理实现成本不同。
- 不做同条件 A/B，会把“后处理没对齐”的问题误判成“模型不行”。

## 为什么要“三层测试”

| 层级 | 回答的问题 | 不做的风险 |
| --- | --- | --- |
| 模型层（A/B） | 该选哪个模型组合 | 选型靠感觉，后续返工大 |
| 边缘运行层 | 真实相机+推理+缓存是否稳定 | 线上掉帧、内存顶满、事件丢失 |
| 系统验收层 | 是否达成计划书能力 | 功能看似齐全但真实场景不可用 |

## 与计划书能力的直接映射

| 计划能力 | 最小必测项 |
| --- | --- |
| 当前场景理解 | run-once 正确输出对象与置信度 |
| 历史事件可追溯 | event 入库、查询、去重 |
| 状态判断（object/zone） | freshness/stale/fallback 路径验证 |
| 设备可控（snapshot/clip） | 命令成功率与延迟 |
| 安全与审计 | 敏感动作在 `audit_logs` 可追踪 |

## 跳过测试的代价
- 误选模型：性能与质量不达标，返工转换与后处理。
- 误判稳定性：压测前看似正常，实际运行出现抖动与丢事件。
- 误判交付完成：文档齐全但行为不可验收。

## 建议执行顺序
1. 先补齐真实链路先决条件（真实输入、真实后处理）。
2. 再做模型 A/B（A1/A2/B1/C1/C2）。
3. 最后做系统验收映射（能力到测试一一对应）。
